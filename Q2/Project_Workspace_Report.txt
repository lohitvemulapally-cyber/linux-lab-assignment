=== Question 2: Secure Project Workspace Setup ===

1. Directory Structure Created
Command: mkdir -p ProjectAlpha/{docs,scripts,data}
Output:
total 20
drwxr-xr-x 5 lohit lohit 4096 Jul  3 06:44 .
drwxr-xr-x 3 lohit lohit 4096 Jul  3 06:44 ..
drwxr-xr-x 2 lohit lohit 4096 Jul  3 06:44 data
drwxr-xr-x 2 lohit lohit 4096 Jul  3 06:44 docs
drwxr-xr-x 2 lohit lohit 4096 Jul  3 06:44 scripts
Observation: Created a ProjectAlpha directory containing three subfolders (docs, scripts, data) to organize project files by purpose.

2. File Permissions Before and After Modification
Command (before): touch ProjectAlpha/docs/plan.txt; ls -l ProjectAlpha/docs/plan.txt
Output (before): -rw-r--r-- 1 lohit lohit 0 Jul  3 06:45 ProjectAlpha/docs/plan.txt
Observation (before): The file was created with default permissions 644 (rw-r--r--), derived from the base file permission 666 minus the umask value of 022.

Command (after): chmod 750 ProjectAlpha/docs/plan.txt; ls -l ProjectAlpha/docs/plan.txt
Output (after): -rwxr-x--- 1 lohit lohit 0 Jul  3 06:45 ProjectAlpha/docs/plan.txt
Observation (after): Permissions were tightened to 750, giving the owner full access, the group read/execute access, and no access to others, which secures sensitive project files from unauthorized users.

3. Ownership Details
Command: ls -l ProjectAlpha/docs/plan.txt; stat ProjectAlpha/docs/plan.txt
Output:
File: ProjectAlpha/docs/plan.txt
Size: 0        Blocks: 0        IO Block: 4096  regular empty file
Device: 8,48    Inode: 44942    Links: 1

Observation: Both the owner and group of the file are "lohit" (UID/GID 1000). stat also reveals the inode number (44942), confirming the file's unique identity on the filesystem independent of its name.

4. Umask Value Used
Command: umask
Output: 0022
Observation: The default umask of 022 subtracts write permission for group and others, resulting in files defaulting to 644 and directories to 755.

Additional demonstration - Custom umask:
Command: umask 027; touch ProjectAlpha/scripts/deploy.sh; ls -l ProjectAlpha/scripts/deploy.sh
Output: -rw-r----- 1 lohit lohit 0 Jul  3 06:46 ProjectAlpha/scripts/deploy.sh
Observation: Changing the umask to 027 caused the new file to be created with 640 permissions instead of the default 644, proving that umask directly controls default access rights at file creation time.

5. Explanation of How Permissions Protect Project Data
Linux file permissions (read, write, execute) combined with ownership (user and group) form the core access control mechanism for the filesystem. By restricting a file like plan.txt to 750, only the owner can modify it, only members of the owning group can read/execute it, and all other users are completely denied access. This prevents unauthorized users on a shared multi-user system from viewing or tampering with sensitive project documentation. The umask further reinforces this by ensuring every newly created file starts with a secure baseline permission, reducing the risk of accidentally leaving new files world-writable or world-readable.
